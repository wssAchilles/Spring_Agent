### B1. TokenCounter 线程安全修复

**验证结果**：✅ 已验证，原方案目标文件有误，实际问题在 TokenCounter.java

**问题定位**：
- `backend/qknow-hermes/.../cost/TokenCounter.java:27-31`
- `ConcurrentHashMap.merge()` 中的 remapping 函数执行了非原子的 get+set 操作：
  ```java
  existing.setPromptTokens(existing.getPromptTokens() + added.getPromptTokens());
  existing.setCompletionTokens(existing.getCompletionTokens() + added.getCompletionTokens());
  ```
- `TokenUsage.java` 本身是 21 行 Lombok POJO，无 add() 方法

**执行步骤**：
1. 读取 `TokenCounter.java`，定位第 27-31 行的 merge lambda
2. 方案一（推荐）：将 `TokenUsage` 的字段改为 `AtomicLong`，add 操作使用 `addAndGet()`
3. 方案二：在 merge lambda 中加 `synchronized` 块
4. 编写并发测试：多线程 1000 次累加后值准确

**验收标准**：
- [ ] 并发测试通过
- [ ] TokenUsage 字段类型为 AtomicLong 或 merge 操作线程安全

## 阶段 C：RAG 质量提升（Week 2-4）

### C1. 查询增强管线 — HyDE + Multi-Query

**验证结果**：✅ 已验证
- `backend/qknow-module-kmc/.../service/rag/QueryTransformService.java` 存在（104 行）
- 已有 `rewriteQuery()` 和 `compressQuery()` 方法
- `RagRetrievalService.java`（220 行）是主编排器
- `IChatClientService` 接口用于获取 ChatClient（非直接使用 `ChatClient.Builder`）

**参考来源**：
- LangChain MultiQueryRetriever：https://python.langchain.com/docs/modules/data_connection/retrievers/MultiQueryRetriever
- LlamaIndex HyDE：https://docs.llamaindex.ai/en/stable/examples/query_transformations/HyDEQueryTransform.html
- **【已验证】arXiv 2026-04 论文**：HyDE 在数值型 QA 中适得其反（R@5=0.544 vs Dense=0.587）。**必须对包含数字/日期/ID 的查询禁用 HyDE**。
- **【已验证】短板效应**：弱检索路径拖低整体精度。应动态排除分数过低的检索路径。

**执行步骤**：
1. 读取 `QueryTransformService.java` 全文，理解现有 `rewriteQuery()` 和 `compressQuery()` 的实现模式
2. 新增 `generateHypotheticalDocument(String query)` 方法：
   - 使用 `IChatClientService` 获取 ChatClient
   - prompt："Given a question, write a 3-5 sentence plausible answer paragraph. Return ONLY the paragraph."
   - **关键**：增加查询类型检测，对包含数字模式（`\d+`, `Day\d+` 等）的查询禁用 HyDE
3. 新增 `expandQueries(String query, int count)` 方法：
   - prompt："Generate N semantically diverse reformulations. Return as JSON array."
   - 将原始 query 加入列表首位
4. 修改 `RagRetrievalService.retrieve()`，在检索前根据 `hermes.rag.query-enhance.strategy` 配置调用
5. 在 `CandidateFusionService` 中增加弱路径排除：某路 top-1 结果 score < 0.3 时排除该路
6. 新增配置段：
   ```yaml
   hermes:
     rag:
       query-enhance:
         enabled: true
         strategy: multi_query  # hyde / multi_query / none
         variant-count: 3
   ```

**验收标准**：
- [ ] `QueryEnhancer` 或等效方法存在于 `QueryTransformService.java`
- [ ] 配置 `strategy: none` 时行为与优化前完全一致
- [ ] 数值型查询不触发 HyDE
- [ ] 黄金测试用例通过
