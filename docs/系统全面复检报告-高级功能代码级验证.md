## 二、RAG 引擎深度复检

### 2.1 核心管线架构

```
用户查询
  │
  ├─ 1. QueryRouter.classify() → SIMPLE? → 跳过检索
  │
  ├─ 2. 权限检查 (PermissionFilter)
  │
  ├─ 3. retrieveOnce() — 并行 4 路检索：
  │     ├─ VectorRetriever (语义, top50)
  │     ├─ KeywordRetriever (全文, top50)
  │     ├─ MetadataRetriever (元数据, top20)
  │     └─ GraphRagRetriever (图谱, top50)
  │
  ├─ 4. CandidateFusionService.fuse() → RRF(k=60)
  │
  ├─ 5. RagRerankService.rerank() → ColBERT粗排 + DashScope精排
  │
  ├─ 6. RagContextBuilder.buildContext() → 上下文构建(12KB限制)
  │
  └─ 7. CragRetrievalEvaluator.evaluate() → 自纠正循环
```

## 十一、复检结论

### 系统整体评估

系统核心功能（RAG 检索、Agent 对话、SSE 流式、对话持久化）**运行正常**，已通过浏览器联调验证。

### 关键风险

1. **线程池管理**: RAG 管线每次请求创建新线程池，Plan-and-Solve 使用无界公共线程池，高并发下存在崩溃风险。
2. **中文处理**: ColBERT 粗排、fail-plausible 检测对中文文本无效。
3. **评估成本**: RAGAS/RAGChecker 的 LLM 调用次数随数据量线性增长，无并行化。
4. **前端占位**: memory_recall 详情、Faithfulness 指标、召回测试设置持久化均为未完成的占位实现。

### 建议修复顺序

1. **P0**: R1（线程池）、F1（设置持久化）、F2（memory_recall 渲染）
2. **P1**: A5（中文 Jaccard）、W1（checkpoint 删除）、E1/E4（评估并行化）
3. **P2**: R2（死代码清理）、R3（中文 ColBERT）、M1（consolidation 阈值）
