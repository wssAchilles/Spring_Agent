**修复步骤**：
1. 连接 PostgreSQL

3. 验证菜单已注册

**验证命令**：

```sql
SELECT menu_id, menu_name, path, component FROM system_menu WHERE menu_id = 2430;
-- 预期：2430 | LLM 可观测性 | observability | kd/observability/index
```

**注意**：不要写代码，只执行 SQL。

## 问题 2：/hermes/approval/pending 返回 404

**根因**：`ApprovalController` 在 `qknow-hermes-core` 模块中，但主应用 `QKnowApplication` 排除了 `tech.qiantong.qknow.hermes..*` 包扫描。Hermes 作为独立微服务运行，其 controller 不在主应用上下文中。

**修复方案**：在 `qknow-module-kb` 模块中创建一个薄 HTTP controller，代理 Hermes 审批端点。

**具体步骤**：
1. 在 `backend/qknow-module-kb/qknow-module-kb-biz/src/main/java/tech/qiantong/qknow/module/kb/controller/admin/` 下新建 `ApprovalProxyController.java`
2. 注入 `tech.qiantong.qknow.hermes.flow.node.ApprovalNodeExecutor`（通过 hermes-core 依赖）
3. 暴露三个端点：
   - `GET /hermes/approval/pending` → 调用 `ApprovalNodeExecutor.getPendingApprovals()`
   - `POST /hermes/approval/approve` → 调用 `ApprovalNodeExecutor.approve()`
   - `POST /hermes/approval/reject` → 调用 `ApprovalNodeExecutor.reject()`
4. 如果依赖链不干净（hermes-core 不能被 kb-biz 依赖），则隐藏前端审批入口，等后端端点可用后再启用

**验证命令**：
```bash
curl -s http://localhost:8099/hermes/approval/pending | jq '.'
# 预期：返回 {"code":200,"data":[]}
```

## 问题 3：社区检测接口业务失败

**根因**：`GraphCommunityService` 调用 `gds.graph.project.cypher()`，但 Neo4j 容器未安装 Graph Data Science (GDS) 插件。

**修复方案**：两个选择（二选一）：

**方案 A：安装 GDS 插件（推荐）**
1. 修改 `docker-compose.yml` 的 Neo4j 服务，添加 GDS 插件：
```yaml
neo4j:
  image: neo4j:5.26.0
  environment:
    NEO4J_AUTH: neo4j/${NEO4J_PASSWORD:-neo4jpass123}
    NEO4J_PLUGINS: '["graph-data-science"]'
  volumes:
    - neo4j_data:/data
    - neo4j_logs:/logs
    - neo4j_plugins:/plugins
```
2. 添加 `neo4j_logs` 和 `neo4j_plugins` 卷
3. 重启 Neo4j：`docker compose down neo4j && docker compose up -d neo4j`

**方案 B：后端降级处理（更安全）**
1. 修改 `GraphCommunityService.detectCommunities()` 方法
2. 在调用 GDS 之前检查插件是否可用
3. 如果 GDS 不可用，返回明确的 501 错误："Neo4j GDS 插件未安装，社区检测功能不可用"
4. 前端显示友好提示而非内部异常

**验证命令**：
```bash
# 方案 A 验证 GDS 安装
docker exec agent-neo4j cypher-shell -u neo4j -p neo4jpass123 "CALL gds.version()"

# 方案 B 验证降级
curl -s -X POST http://localhost:8099/kg/graph/communities/detect | jq '.code'
# 预期：501 + 明确错误信息
```

## 问题 4：Element Plus appendTo 警告

**根因**：`el-dialog` 组件的 `:append-to` 传了 `HTMLDivElement` 对象，Element Plus 当前版本期望 `String` 类型的 CSS selector。

**修复步骤**：
1. 全仓搜索 `append-to`：
```bash
grep -rn "append-to\|appendTo" frontend/src/ --include="*.vue" | head -10
```
2. 将 `:append-to="$refs['xxx']"` 改为 `append-to="body"` 或直接移除 `append-to` 属性（使用默认挂载）

**验证**：打开工具管理页，浏览器控制台无 appendTo 警告。

## 问题 5：size="mini" 已废弃

**根因**：Element Plus 2.7.6 中 `size="mini"` 不再是合法值，应改为 `size="small"`。

**修复步骤**：
1. 全仓搜索：
```bash
grep -rn 'size="mini"' frontend/src/ --include="*.vue" | wc -l
```
2. 批量替换：
```bash
find frontend/src -name "*.vue" -exec sed -i 's/size="mini"/size="small"/g' {} \;
```
3. 验证：
```bash
grep -rn 'size="mini"' frontend/src/ --include="*.vue" | wc -l
# 预期：0
```

**验证**：前端构建通过，控制台无 size 警告。

## 问题 6：Agent 消息发送未验证

**根因**：验证时未触发 Agent 发送消息（需调用外部 LLM，可能产生费用）。

**修复步骤**：
1. 登录系统，进入 Bot 管理 → 选择一个 Bot → 进入 Agent 调试页面
2. 发送一个低成本问题（如"你好"）
3. 观察：
   - 是否出现 `/kb/conversation` 或 `/kb/agent/testChatMessages` 接口调用（Network 面板）
   - 响应是否 200
   - 消息列表是否正确渲染

**注意**：这需要后端 + Hermes + LLM API 全链路可用。如果 LLM API 不可用，只验证到接口 200 即可。
