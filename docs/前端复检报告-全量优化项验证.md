## 九、复检发现的问题

### 问题 1: VectorRetriever NumberFormatException

| 项目 | 内容 |
|------|------|
| **严重度** | 🔴 高 |
| **影响范围** | 所有使用向量检索的知识库 |
| **发现方式** | 召回测试返回 0 结果 + 后端日志报错 |
| **错误信息** | `java.lang.NumberFormatException: For input string: ""` |
| **发生位置** | `VectorRetriever.java:50` |

#### 问题原因

`KmcKnowledgeBaseDO.embeddingModelProvider` 字段为空字符串 `""`，代码直接调用 `Long.valueOf("")` 导致崩溃。

```java
// 修复前（崩溃）
EmbeddingModel embeddingModel = aiModelService.getEmbeddingModel(
    Long.valueOf(kb.getEmbeddingModelProvider()), kb.getEmbeddingModel());
```

#### 解决方案

在 `Long.valueOf()` 前添加空值检查，为空时回退到默认 provider ID：

```java
// 修复后（防御性）
String providerStr = kb.getEmbeddingModelProvider();
if (providerStr == null || providerStr.isBlank()) {
    log.warn("embeddingModelProvider is empty for knowledgeBaseId={}, falling back to default", knowledgeBaseId);
    providerStr = "1";
}
EmbeddingModel embeddingModel = aiModelService.getEmbeddingModel(
    Long.valueOf(providerStr), kb.getEmbeddingModel());
```

#### 修复文件

`backend/qknow-module-kmc/qknow-module-kmc-biz/src/main/java/tech/qiantong/qknow/module/kmc/service/rag/VectorRetriever.java`

#### 验证结果

修复后向量召回从 0 恢复到 **17 条结果**。

### 问题 2: LangFuse RAGAS 评分写入返回 400

| 项目 | 内容 |
|------|------|
| **严重度** | 🟡 中 |
| **影响范围** | 可观测性面板的质量评分 |
| **发现方式** | Hermes 日志 `LangFuse score recording failed: status=400` |
| **错误信息** | LangFuse API 返回 HTTP 400 |
| **发生位置** | `LangFuseTracingService.java:131-153` |

#### 问题原因

1. 评分 JSON 缺少 `dataType: "numeric"` 字段
2. `value` 使用 `%f` 格式化，产生 `0.850000` 这样的多余小数位
3. 评分名称可能包含 LangFuse 不支持的字符

```java
// 修复前（400 错误）
String body = String.format("""
    {"id":"%s","traceId":"%s","name":"%s","value":%f,"source":"api"%s}
    """, ...);
```

#### 解决方案

1. 添加 `dataType: "numeric"` 字段
2. 使用 `%s` 替代 `%f`，避免精度问题
3. 评分名称安全处理：`name.replaceAll("[^a-zA-Z0-9_-]", "_")`
4. 添加 429 重试逻辑（3 次指数退避）

```java
// 修复后
String safeName = name.replaceAll("[^a-zA-Z0-9_-]", "_");
String body = String.format("""
    {"id":"%s","traceId":"%s","name":"%s","value":%s,"source":"api","dataType":"numeric"%s}
    """, java.util.UUID.randomUUID(), traceId, safeName, value, ...);
```

#### 修复文件

`backend/qknow-hermes/qknow-hermes-core/src/main/java/tech/qiantong/qknow/hermes/observability/LangFuseTracingService.java`

#### 验证结果

修复后不再出现 400 错误日志。

### 问题 3: LangFuse API 429 限流

| 项目 | 内容 |
|------|------|
| **严重度** | 🟢 低 |
| **影响范围** | 可观测性面板的 traces 获取 |
| **发现方式** | 前端 API 响应 `{"error":"LangFuse API 返回 429"}` |
| **错误信息** | LangFuse API 返回 HTTP 429 (Too Many Requests) |
| **发生位置** | `HealthCheckController.java` + `LangFuseTracingService.java` |

#### 问题原因

所有 LangFuse HTTP API 调用（trace/generation/score/traces/observations）均未实现重试逻辑，遇到 429 直接失败。

#### 解决方案

为所有 LangFuse API 调用添加指数退避重试：

| 调用 | 重试次数 | 退避策略 |
|------|---------|---------|
| trace 创建 | 3 次 | 500ms × (retry+1) |
| generation 记录 | 3 次 | 500ms × (retry+1) |
| score 记录 | 3 次 | 500ms × (retry+1) |
| traces 获取 | 3 次 | 1000ms × (retry+1) |
| observations 获取 | 2 次 | 500ms × (retry+1) |

#### 修复文件

- `backend/qknow-hermes/.../observability/LangFuseTracingService.java`（3 处）
- `backend/qknow-module-kb/.../controller/admin/HealthCheckController.java`（2 处）

#### 验证结果

代码已添加重试逻辑，LangFuse 限流恢复后将自动重试成功。

### 问题 4: Neo4j 图谱召回为 0

| 项目 | 内容 |
|------|------|
| **严重度** | 🟡 中 |
| **影响范围** | 知识图谱检索、社区检测 |
| **发现方式** | 召回测试调试面板「图谱召回=0」+ 页面提示 Neo4j 连接失败 |
| **错误信息** | `Unable to connect to 127.0.0.1:7687` |
| **发生位置** | Neo4j 容器未启动 |

#### 解决方案

启动 Docker 后运行 Neo4j 容器：

```bash
docker compose up -d neo4j
```

#### 验证结果

Neo4j 容器启动后连接正常：

```
docker exec agent-neo4j cypher-shell -u neo4j -p neo4jpass123 "RETURN 1"
→ 1
```
