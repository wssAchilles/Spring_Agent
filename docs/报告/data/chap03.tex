\section{gRPC 通信协议与底层网络实现}

控制面与认知面摒弃了传统的 HTTP RESTful 协议，转而深度定制了基于 HTTP/2 的 gRPC 框架进行跨进程通信。在底层网络传输层，系统采用了 Netty 作为 NIO 异步通信框架，并开启了 \texttt{Epoll} 原生传输模式以最大化利用 Linux 内核的网络 I/O 性能。Protobuf 服务契约（.proto）严格定义了三个核心 RPC 存根（Stub）：

\texttt{Chat} 方法采用服务端流式响应模式（Server-side Streaming）。控制面通过 \texttt{ChatRequest} 一次性装载高密度的环境数据（包含环境提示词、多轮对话历史、K-NN 预检索的向量文本块及工具回调地址）；认知面则通过持续开启的 Stream 管道异步推送 \texttt{ChatEvent} 事件流。为了应对复杂推理场景中大语言模型可能输出的异构数据，\texttt{ChatEvent} 巧妙地运用了 Protobuf 的 \texttt{oneof} 联合体语法，精确隔离了 8 种互斥的子事件：如用于打字机效果的 \texttt{StreamingChunk}（毫秒级推送增量 Token）、用于中途暴露思维链的 \texttt{ReflectionStart} 与 \texttt{ToolInvoked}，以及标记推演结束的 \texttt{DoneSignal}。

\texttt{ExecuteFlow} 同样为流式接口，专为 DAG 异步计算设计；而 \texttt{HealthCheck} 则是基于单播（Unary）的一元调用，配合 Consul 注册中心实现每 5 秒一次的心跳探活与服务剔除。

在工程实践中，为解决 gRPC 长连接在云原生环境（如 LVS 负载均衡器下）容易被静默切断的问题，系统在 \texttt{ManagedChannel} 级别全局启用了 Keep-Alive 探针机制（配置为每 30 秒发送一次 PING 帧）。同时，控制面实现了指数退避（Exponential Backoff）的自动重试拦截器。最核心的设计哲学在于：控制面揽下了所有如鉴权解析、数据库 I/O 等“脏活累活”，使得发往 gRPC 管道的数据完全自洽，从而确保了认知面的绝对无状态化。

\subsection{文档切块实现}

RecursiveSplitter 的核心实现采用递归下降算法。分隔符优先级数组定义为：

\begin{verbatim}
DEFAULT_CHINESE_SEPARATORS = ["\n\n", "\n", "。", "；", "！", "？",
                               "!", "?", "，", ",", " ", ""]
\end{verbatim}

递归函数 \texttt{recursiveSplit} 的逻辑为：尝试使用当前分隔符切分文本；若切分后的片段仍超过 chunkSize，递归使用下一级分隔符；当所有分隔符都尝试完毕后，回退到字符级硬切分。

重叠应用函数 \texttt{applyOverlap(chunks)} 在相邻块之间添加重叠：从前一个块的尾部取 \texttt{chunkOverlap} 个字符，拼接到当前块的头部。

父子切块函数 \texttt{splitParentChild} 先以大块切分生成父块，再对每个父块以小块切分生成子块，子块通过 \texttt{parent\_segment\_id} 元数据关联父块。

\subsection{向量存储实现}

向量存储使用 Spring AI 的 \texttt{PgVectorStore}，通过 \texttt{ConcurrentHashMap} 缓存实例，键为 \texttt{embeddingModel.getClass().getName() + "@" + identityHashCode}。构建代码为：

\begin{verbatim}
PgVectorStore.builder(jdbcTemplate, embeddingModel).build()
\end{verbatim}

嵌入模型工厂 \texttt{EmbeddingServiceImpl} 支持三种平台：OpenAI（标准 API）、TongYi/DashScope（兼容模式端点 \texttt{dashscope.aliyuncs.com/compatible-mode}）和 Ollama（本地部署）。

\subsection{检索服务实现}

\texttt{RagRetrievalService} 是检索引擎的主入口，其 \texttt{retrieve()} 方法实现了完整的检索流程：

（1）查询路由：对 SIMPLE 级别的查询跳过检索。

（2）权限检查：基于 RBAC 的知识库访问控制。

（3）首次检索：调用 \texttt{retrieveOnce()} 执行四路并行召回。

（4）CRAG 评估：对首次检索结果进行质量分类。

（5）条件二次检索：当评估结果为 INCORRECT 且提供了重写查询时，使用重写查询重新检索。

\texttt{retrieveOnce()} 方法使用 4 线程池并行执行四条检索路径，每条路径的超时时间为 30 秒。候选 top-K 设置为 $\max(\text{topK} \times 3, 50)$。

\subsection{上下文构建实现}

\texttt{RagContextBuilder} 负责将检索结果组装为 LLM 可用的上下文文本。构建流程为：

（1）父段扩展：如果子块有 \texttt{parentSegmentId}，从数据库中获取父段内容并前置。

（2）相邻段扩展：如果没有父段，则获取 position-1 和 position+1 的相邻段。

（3）内容去重：按 \texttt{content.hashCode()} 去重。

（4）Token 估算：ASCII 单词计 1 个 Token，非 ASCII 非空白字符计 2 个 Token。

（5）预算截断：当总字节数超过 12000 或 Token 数超过阈值时停止追加。

输出格式为：
\begin{verbatim}
[来源 {index}] {docName} / segmentId={id} / parentId={parentId}
内容：{content}
\end{verbatim}

\section{Hermes 认知内核实现}

\subsection{反思循环实现}

\texttt{HermesKernel.reflect()} 方法实现了完整的反思循环。关键实现细节包括：

（1）提示词重构：首次尝试仅包含系统提示词和参考知识；后续尝试附加完整的评分反馈向量（事实性分数、相关性分数、指令遵循度分数、综合分数和改进建议）。

（2）自洽性检查：通过 3 次独立 LLM 调用获取采样结果，使用空白字符分词后计算成对 Jaccard 相似度。空响应或纯空白响应被过滤，有效采样数不足 2 时默认返回一致性分数 1.0。

（3）双重门限：通过条件为 \texttt{judgeResult.passed AND consistency >= 0.6}。

\subsection{Plan-and-Solve 实现}

\texttt{planAndSolve()} 方法的实现包括：

（1）复杂度判定：\texttt{isComplexQuestion()} 检查查询长度是否超过 100 字符或是否包含预定义的复杂关键词（"对比"、"比较"、"分析"、"总结"等 10 个词）。

（2）任务分解：LLM 返回 JSON 数组，每个元素包含 \texttt{taskId}、\texttt{objective} 和 \texttt{dependencies}。

（3）环检测：使用 DFS 着色算法（白/灰/黑模式）检测依赖环。

（4）并行执行：按拓扑顺序逐层执行，无依赖任务通过 \texttt{Completable\-Future} 异步并行执行，结果通过 \texttt{LinkedHashMap} 保持顺序。

\subsection{记忆系统实现}

短期记忆支持双模式存储：内存模式使用 \texttt{ArrayList<Message>}，Redis 模式使用 \texttt{IRedisService} 的 List 操作（键格式 \texttt{memory:short:\{sessionId\}}）。摘要压缩使用 LLM 对早期对话生成摘要，替换为单条 \texttt{SystemMessage}。

长期记忆的检索流程：先从向量存储中获取 $\max(\text{topK} \times 3, 20)$ 条结果，然后按复合评分公式重新排序。记忆合并的阈值为余弦相似度 0.85，合并时更新元数据中的 \texttt{updated\_at} 和 \texttt{consolidated\_count}。

\section{DAG 工作流引擎实现}

\subsection{节点执行模板}

\texttt{BaseNodeBO.execute()} 实现了模板方法模式的框架：

（1）\texttt{prepareInput(context)}：从节点定义的 input JSON 中读取变量名，从上下文变量中获取对应值。

（2）\texttt{executeLogic(inputData, context)}：子类实现的具体执行逻辑。

（3）\texttt{processOutput(result, context)}：将结果输出合并到上下文变量。

（4）\texttt{findNextNodes(result, context)}：根据边列表查找后继节点。

变量插值函数 \texttt{format(text, context)} 使用正则 \texttt{\{\{\textbackslash s*(.+?)\textbackslash s*\}\}} 匹配占位符，从上下文中查找替换值。

\subsection{条件分支实现}

\texttt{ConditionNodeBO} 实现了内置的表达式评估器（不依赖 SpEL），支持 \texttt{==}、\texttt{!=}、\texttt{>}、\texttt{<}、\texttt{>=}、\texttt{<=} 比较操作。评估顺序为：先尝试数值比较（\texttt{Double.parseDouble}），失败则回退到字符串比较（\texttt{String.compareTo}），最后尝试布尔比较。

条件匹配采用"首个匹配胜出"策略：按配置顺序评估条件，第一个满足的条件决定路由分支。\texttt{default} 条件始终匹配但不中断循环（最后一个 default 生效）。

\subsection{流式输出实现}

\texttt{ReplyNodeBO} 实现了三种输出模式：

（1）ChatFlow 模式：解析内容模板中的 \texttt{\{\{ nodeUuid.variable \}\}} 引用，将 LLM 流式段作为 \texttt{Flux<String>} 透传，非 LLM 段作为单元素 Flux 发射。

（2）工作流流式模式：检查源节点类型是否为 LLM 且存在 Flux，存在则包装 JSON 元数据后透传，否则从上下文读取值。

（3）阻塞模式：直接从上下文变量读取值，组装输出 Map。

\subsection{Human-in-the-loop 人工审批与挂起恢复实现}

在企业级 AI 编排中，敏感操作（如大额转账、邮件群发）必须引入人类介入（Human-in-the-loop）。\texttt{ApprovalNodeExecutor} 摒弃了传统的自旋锁等待方案，转而采用了更为优雅的异步挂起/唤醒（Suspend/Resume）机制。在内存层，系统利用静态 \texttt{ConcurrentHashMap<String, CompletableFuture<Boolean>>} 作为信号量注册中心来映射唯一的 \texttt{TraceID}。

当流程流转至审批节点时，执行线程会主动陷入挂起状态，通过带有超时上限（如 24 小时）的 \texttt{CompletableFuture.get(timeout)} 出让 CPU 时间片。同时，系统通过消息队列向企微/钉钉推送 Webhook 审批表单卡片。一旦人工在前端点击审批通过或驳回，控制面的 REST API 回调端点将极速寻址到对应的 \texttt{TraceID}，并通过 \texttt{future.complete(true/false)} 瞬间唤醒沉睡的 DAG 工作流线程。若在超时窗口内未收到任何人工回调，触发器则会抛出 \texttt{TimeoutException}，迫使流程安全回滚并走向降级分支。

\section{前端实现}

\subsection{DAG 画布与前端工程化实现}

前端 DAG 可视化画布深度二次封装了 VueFlow 引擎。为了保障数千个节点下画布的流畅渲染，系统开启了 VueFlow 的按需渲染（Only Render Visible Elements）特性，配合硬件 GPU 加速，实现了复杂工作流拓扑下维持 60 FPS 的丝滑体验。画布的工程配置参数被严格校准：对齐网格粒度设定为 20px，默认缩放设为 0.2，视口极限缩放区间限制在 $[0.1, 2.0]$，节点间的连接线统一采用平滑的贝塞尔曲线（Smooth Step / Bezier）。

在组件化设计上，系统抽象了 6 种高度定制的业务节点类型。基于 Vue 3 的 Composition API 和 Pinia 状态树，每个节点组件内部实现了精准的状态响应式隔离。每个视觉节点标准配备了左侧的 \texttt{Target Handle}（输入锚点）和右侧的 \texttt{Source Handle}（输出锚点），并在节点边缘动态挂载了用于快速新建分支的“+”按钮以及一键销毁按钮。当用户点击特定节点时，通过 Vue 的全局事件总线（EventBus），右侧的侧边栏抽屉（Drawer）会动态挂载对应节点类型的配置表单组件，实现了“图谱拓扑展示”与“业务属性编辑”的完美前端解耦。

\subsection{流式通信实现}

前端使用 \texttt{@microsoft/fetch-event-source} 库实现 SSE 流式通信。请求头包含 \texttt{Content-Type: application/json} 和 \texttt{Authorization: Bearer <token>}。事件处理支持三种类型：文本内容（增量追加到 Markdown 渲染器）、工具调用（折叠卡片显示输入输出 JSON）和记忆召回（显示召回数量和相似度分数）。

\section{安全实现}

\subsection{无状态 JWT 认证与防重放实现}

平台的登录认证架构摒弃了传统的 Session/Cookie 机制，采用了深度定制的无状态 JWT（JSON Web Token）方案，以适应微服务与 Kubernetes 的横向扩展需求。

\textbf{登录阶段（Login）}：客户端通过加密通道发送包含图形验证码校验的登录载荷。服务端在拦截器层拦截后，调用 Spring Security 的内部校验器，使用强哈希算法（BCrypt）对比密码盐值。校验通过后，系统并非直接将用户敏感信息塞入 JWT Payload，而是生成一个无实际语义的 \texttt{UUID} 作为会话密钥，将完整的 \texttt{LoginUser} 实体序列化并下沉存储至高可用 Redis 集群中（设定硬性 TTL 为 600 分钟）。随后，系统使用 HS512 算法对包含该 \texttt{UUID} 的 JWT 进行签名下发。这种“JWT 令牌 + Redis 会话”的双重设计，完美解决了原生 JWT 无法在服务端主动注销（Revoke）的安全死穴。

\textbf{请求鉴权阶段（Authentication）}：系统在全局拦截层注册了专属的 JWT 鉴权过滤器链。对于每个 HTTP 请求，鉴权过滤器首先从请求头的 \texttt{Authorization: Bearer} 字段提取 JWT 密文，利用公钥验签并逆向解析出 \texttt{UUID}。随后以此为键向 Redis 发起 $O(1)$ 的高速查询。若命中，则将反序列化后的安全上下文（SecurityContext）注入当前线程。为提升用户体验，系统还引入了滑动窗口续期机制（Sliding Window Session）：当查询 Redis 发现当前会话的剩余存活时间不足 20 分钟时，系统会在底层自动刷新 Redis 过期时间。

\subsection{RBAC 授权}

权限检查通过 \texttt{@PreAuthorize} 注解实现，全系统共 410+ 处注解。管理员（userId == 1）拥有通配符权限。数据范围过滤通过 \texttt{@DataScope} 切面注入 SQL WHERE 子句，支持全部、自定义、本部门、部门及下级、仅本人五种范围。
